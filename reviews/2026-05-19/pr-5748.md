## Verdict: NEEDS_DISCUSSION

## Summary

Adds an "Auto" theme option to the Settings theme picker that follows the OS light/dark preference in real time. The feature works correctly end-to-end: `useCurrentAppTheme()` resolves 'auto' → light/dark via the reactive `usePrefersColorScheme()` hook, `getThemeName()` handles 'auto' in localStorage for the static path, and `ThemePreview` renders a split half-light/half-dark preview for the auto tile. Good test coverage in `themes.test.ts`. Two issues need to be addressed before merge: a missing storyshot update and the PR title not following the project commit convention.

## Findings

### Warnings
- **[HIGH]** `frontend/src/components/App/defaultAppThemes.ts:163` — Adding `autoTheme` to `defaultAppThemes` changes the Settings page theme-picker rendering, but `frontend/src/components/App/Settings/__snapshots__/Settings.General.stories.storyshot` was not updated. If the project runs storyshot tests this will produce a snapshot mismatch. Run `npm run storyshots -- -u` to regenerate.

- **[MEDIUM]** PR title "Automatically switch theme based on OS theme" does not follow the `area: Component: Description` project convention (e.g. `frontend: Settings: Add auto OS theme`). The maintainer requests commit-guideline fixes before approving.

### Suggestions
- **[MEDIUM]** `frontend/src/App.tsx:53` — `osColorScheme` is included in the `useMemo` dep array for `muiTheme`, but `currentAppTheme` (returned by `useCurrentAppTheme()`) is itself reactive to OS changes — when the OS theme changes, `usePrefersColorScheme()` fires, `useCurrentAppTheme()` returns a new `lightTheme`/`darkTheme` object reference, and the memo already re-runs via the `currentAppTheme` dep. The `osColorScheme` dep is harmless but redundant and could be removed along with the `// eslint-disable-next-line react-hooks/exhaustive-deps` suppressor.

- **[LOW]** `frontend/src/lib/themes.ts:566` — `getOsColorScheme()` duplicates the synchronous `window.matchMedia('(prefers-color-scheme: dark)').matches` check that already exists in `usePrefersColorScheme()`. Consider extracting it as a shared helper, or have `getThemeName` call `usePrefersColorScheme` indirectly (though that would turn `getThemeName` into a hook). As-is the duplication is small but worth noting.

### Positive
- `ThemePreview` correctly renders a split half-light/half-dark preview for the 'auto' tile without ever passing the unresolved `autoTheme` to `createMuiTheme` — clean guard via the `theme.name === 'auto'` branch.
- Four new `getThemeName` test cases in `themes.test.ts` cover OS-dark, OS-light, and backend `defaultDarkTheme`/`defaultLightTheme` overrides for the 'auto' preference, giving good confidence in the static resolution path.
- `autoTheme` is placed first in `defaultAppThemes`, making it the most prominent option — good UX default for users who want OS-following behavior.
