## Verdict: REQUEST_CHANGES

## Summary

Extends `charts/headlamp/values.schema.json` to allow `valueFrom` as an alternative to `value` in the `env` array items, enabling env vars sourced from Secrets and ConfigMaps. The schema logic is correct for the cases covered, but the `valueFrom` definition is incomplete relative to the Kubernetes API surface, and the PR title/commit message does not follow the project's commit convention.

## Findings

### Warnings
- **[HIGH]** `charts/headlamp/values.schema.json` â€” The PR title and commit subject use `feat(helm): ...` (Conventional Commits style), but headlamp uses `area: Component: Description` format (e.g. `helm: env: Add valueFrom support`). The primary maintainer actively requests commit-guideline fixes before approving. This will need to be changed.

- **[MEDIUM]** `charts/headlamp/values.schema.json:284` â€” The `valueFrom` schema only defines `configMapKeyRef` and `secretKeyRef`, both with `additionalProperties: false`. Kubernetes [`EnvVarSource`](https://kubernetes.io/docs/reference/kubernetes-api/workload-resources/pod-v1/#EnvVarSource) also supports `fieldRef` (Pod metadata) and `resourceFieldRef` (container resource limits/requests). With `additionalProperties: false` on the `valueFrom` wrapper, any chart user who passes `fieldRef` or `resourceFieldRef` will get a schema validation error even though the Kubernetes API accepts these. Consider either removing `additionalProperties: false` from the `valueFrom` wrapper or adding the two missing ref types.

### Suggestions
- **[MEDIUM]** `charts/headlamp/values.schema.json` â€” No Helm chart template change accompanies the schema update. While the Helm template already uses `range .Values.env` and passes entries through verbatim, a test case in `charts/headlamp/tests/` (or at minimum a `helm lint` example in the PR description) would confirm the schema validates correctly end-to-end before the chart is published.

### Positive
- Using `oneOf: [{required: ["value"]}, {required: ["valueFrom"]}]` correctly enforces mutual exclusivity: exactly one of `value` or `valueFrom` must be present per env entry, matching the Kubernetes API contract.
