## Verdict: APPROVE

## Summary

`getNumReplicas` is promoted to module scope as a pure function with an explicit parameter, fixing both the closure-capture issue and the unsafe `parseInt` without radix. Lazy `useState(() => getNumReplicas(resource))` correctly preserves the user's edited replica count across live resource updates while `currentNumReplicas = getNumReplicas(resource)` displays the live server-side value. Clean and well-reasoned change.

## Findings

### Suggestions

- **[LOW]** The PR title says "Memoize replica count computation" and the description mentions `React.useMemo` for `currentNumReplicas`, but the final implementation uses a plain call `getNumReplicas(resource)` (no useMemo). The approach evolved to a simpler and equally correct form — memoising a cheap pure extraction from a prop would add unnecessary overhead — but the title/description should reflect the final implementation for clarity.

### Positive

- `parseInt(replicas, 10)` with explicit radix and `Number.isFinite(parsed)` guard is strictly safer than the old `parseInt(resource.spec.replicas)` without radix.
- Module-scope placement makes `getNumReplicas` independently testable without mounting a component.
- `useState(() => getNumReplicas(resource))` lazy initializer correctly freezes the dialog's initial value at open time, so user edits to the replica count field are preserved even if the parent resource object updates while the dialog is open.
- Handles both `string` and `number` values of `resource.spec.replicas` via the `typeof replicas === 'string'` branch — the old code assumed `replicas` was always a string (`parseInt` on a number is technically fine, but explicit handling is cleaner).
