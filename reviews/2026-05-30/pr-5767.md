## Summary

The `useMemo(() => value, [])` anti-pattern for one-shot initialisation has been correctly replaced with `useState(() => value)` for both `selectedValue` and `customValue`. The `dispatch` dependency added to the `useEffect` dep array is correct since `dispatch` is a stable reference in Redux Toolkit but is good practice to include. All three `eslint-disable` suppressions are cleanly removed.

## Findings

### Suggestions

- **[LOW]** PR title has a trailing space: `"Fix react-hooks/exhaustive-deps "`. Cosmetic, but worth trimming in the description or a future PR.

### Positive

- Replacing `useMemo(() => init, [])` with `useState(() => init)` is the correct pattern for one-shot lazy initialization — avoids re-running the initializer on every render that useMemo with changing deps would trigger, and communicates intent clearly.
- Using `defaultValue.includes(val)` directly instead of the intermediate `options` variable is cleaner since `options` itself was already `props.defaultValue || defaultTableRowsPerPageOptions` — the same data.
- Commit messages (`frontend: NumRowsInput: Fix react-hooks/exhaustive-deps` and `frontend: NumRowsInput: Remove unnecessary useMemo and sync state init`) follow the project format correctly.
