## Summary

The PR has evolved through several commits since its last approval. The approach shifted from a `useEffect`-based page reset to React's render-phase derived-state pattern (`if (condition) setPage(0)` in the render body). This is valid тАФ React explicitly supports calling `setState` during render for derived-state adjustments; the component will discard the current render and retry, converging in at most two passes without an infinite loop. Additional improvements include `normalizedData = data ?? null` for safe undefined handling, and refactoring `getSortData` to be a pure function returning a value rather than calling `setDisplayData` as a side effect.

## Findings

### Suggestions

- **[MEDIUM]** PR description says "handle the page boundary correction inside a `useEffect` hook" but the final implementation uses a render-phase `setPage(0)` check instead. The description should be updated to match the actual implementation so reviewers and future maintainers understand the chosen approach. The render-phase pattern is explicitly documented in React as acceptable for this use case.

- **[LOW]** The `useEffect` at line ~253 (`[data, currentData]` deps) still carries `// eslint-disable-next-line` suppression. Now that `normalizedData` replaces `data`, the dep array reads `[normalizedData, currentData]` тАФ worth verifying there is no longer a stale-closure reason for the suppression, and removing the comment if not needed.

### Positive

- `normalizedData = data ?? null` cleanly normalises `undefined` input to `null` so that all downstream null-checks are consistent; `data` never leaks as `undefined` into state.
- Hoisting `filteredData` into `React.useMemo` above the early return is the correct structural fix: it ensures the filtered length is always available for pagination checks regardless of whether `displayData` is null.
- Render-phase `setPage(0)` stabilises in exactly two renders (second render has `page=0`, so `page !== 0` is false and the call is skipped). No infinite loop risk.
- Refactoring `getSortData` to return a sorted array rather than calling `setDisplayData` internally makes the function pure and composable тАФ the caller `setDisplayData(getSortData() || normalizedData)` is now explicit about the state update.
