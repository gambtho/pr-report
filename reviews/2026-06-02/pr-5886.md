## Verdict: NEEDS_DISCUSSION

## Summary

Adds a `Security > Access` page showing the current user's identity, groups, accessible namespaces, and a per-namespace RBAC permission matrix using `SelfSubjectRulesReview`. Core logic is well-structured: `permissionUtils.ts` handles wildcard/named-resource matching correctly with unit tests, the API call is added cleanly to `clusterApi.ts`, and the router/sidebar are wired correctly. The main concerns are missing error-state UI when API calls fail, commit format issues (missing SubArea), no component-level tests, and the scope of changes to `AuthVisible`, `PortForward`, and other shared components that expand the PR's blast radius significantly.

## Findings

### Warnings

- **[HIGH]** `frontend/src/components/access/Permissions.tsx:226` — `getSelfSubjectRulesReview` failures (e.g., 403 Forbidden, 404 on older clusters) are not surfaced to the user. `rulesQuery.error` is silently ignored; the table renders with all resources showing "No" (denied), which looks like the user has no permissions rather than "the check failed". Add an error Alert when `rulesQuery.isError` is true.

- **[MEDIUM]** Commit subjects `frontend: Add RBAC access inspector` and `frontend: Update RBAC access translations` are both missing the SubArea segment. Project convention requires `area: SubArea: Description`, e.g., `frontend: Security: Add RBAC access inspector` and `frontend: i18n: Update RBAC access translations`.

- **[MEDIUM]** `frontend/src/components/access/Permissions.tsx:203` — Initial namespace defaults to `'default'`, which causes an immediate `SelfSubjectRulesReview` API request for `default` before the namespace list loads. If the user's cluster has no `default` namespace, this produces an unhandled API error. The `useEffect` that corrects `namespace` when the list loads fires after the initial query, not before. Consider initialising `namespace` to `null` and only querying once a valid namespace is selected.

### Suggestions

- **[MEDIUM]** No Vitest component tests for `Permissions.tsx`. The `permissionUtils.ts` unit tests are a good start, but testing `Permissions` with `renderWithProviders` + mocked `useQuery` responses would catch integration issues between the API response and the displayed state.

- **[LOW]** `frontend/src/components/common/Resource/AuthVisible.tsx` — The new `deniedFallback` prop and `AuthVisibleResult` type are backwards-compatible additions. However, the changes to `AuthVisible` and shared resource action buttons (`DeleteButton`, `EditButton`, `RestartButton`, `RollbackButton`, `ScaleButton`, `PortForward`) are a significant expansion of this PR's scope. Consider splitting the `AuthVisible`/action-button enhancements into a separate PR for easier review isolation.

- **[LOW]** `frontend/src/lib/k8s/api/v1/clusterApi.ts:74` — `getSelfSubjectRulesReview` uses `const spec = namespace ? { namespace } : {}`. Sending an empty `spec` (`{}`) to `SelfSubjectRulesReview` may not be valid for all cluster configurations. Since the `Permissions` component always provides a namespace, this fallback is not currently reachable, but add a TypeScript assertion or comment to clarify.

### Positive

- `permissionUtils.ts` handles all three wildcard cases (`verbs: ['*']`, `apiGroups: ['*']`, `resources: ['*']`) and named-resource access (`resourceNames`) correctly, with unit tests covering all branches.
- `sortResourcesForPermissions` places the most commonly accessed resources (pods, deployments, etc.) at the top of the table — good UX default.
- `SelfSubjectRulesReview` is the correct Kubernetes API for this use case; using `staleTime: 60s` for rules and `5min` for user info and API discovery is a sensible caching strategy.
- `rulesStatus?.incomplete` alert surfaces the Kubernetes-reported "incomplete" flag, which is important for clusters with many RBAC rules that hit evaluation limits.
