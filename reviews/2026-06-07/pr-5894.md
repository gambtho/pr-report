## Verdict: REQUEST_CHANGES

## Summary

The new SHA refactors the architecture from an `isActivityOpenRef` ref approach to a dedicated `ResourceWatcher` child component that establishes a live WebSocket watch (`ItemClass.useGet`) and pushes updates to the open `EditorDialog` via `Activity.update`. The conflict-detection logic in `EditorDialog.tsx` is sound and unchanged from the previously-approved version. However, the new `ResourceWatcher` component contains an `eslint-disable react-hooks/exhaustive-deps` suppression, which is an explicit project-convention violation — the project policy is to fix these rather than suppress them. The fix is straightforward (wrap the callbacks in `useCallback`).

## Findings

### Warnings

- **[HIGH]** `frontend/src/components/common/Resource/EditButton.tsx:67` — `ResourceWatcher.useEffect` uses `// eslint-disable-next-line react-hooks/exhaustive-deps` to suppress the exhaustive-deps rule. The missing deps are `onClose`, `handleSave`, `setErrorMessage`, and `activityId`. Per project convention (`learnings.md` / AGENTS.md), `eslint-disable` suppressions for this rule are not accepted — the fix is to wrap `handleClose` and `handleSave` in `useCallback` in `EditButton` (with their respective stable deps), then add all four values to the `ResourceWatcher` effect's dependency array. `setErrorMessage` is already a stable `useState` setter, and `activityId` is a constant string for the lifetime of the activity, so only the two handler functions need `useCallback`. Without the fix, a stale `handleSave` closure could be used if `EditButton` re-renders between watch events — the EditorDialog inside the Activity would keep calling the old save handler.

### Suggestions

- **[MEDIUM]** `frontend/src/components/common/Resource/EditButton.tsx` — The new `ResourceWatcher` component and the updated conflict-detection flow in `EditorDialog.tsx` still have no Vitest test coverage (flagged in the 2026-06-02 review as well). At minimum, a unit test for `ResourceWatcher` that simulates `watchedItem` changing and verifies `Activity.update` is called with the correct `item` prop would guard against regressions in the watch-to-editor pipeline.

- **[LOW]** `frontend/src/components/common/Resource/EditButton.tsx:52` — `handleSave` and `setErrorMessage` are passed as props to `ResourceWatcher` but the `onEditorChanged` callback inside the JSX created by the effect is an inline arrow function `() => setErrorMessage('')`. Since this JSX is passed to `Activity.update` (updating Redux state), each `watchedItem` change creates a new function reference for `onEditorChanged`. This is harmless in practice because `setErrorMessage` is stable, but wrapping the inner arrow function in `useCallback` or hoisting it would make the intent clearer.

### Positive

- The `ResourceWatcher` pattern is a genuine architectural improvement over `isActivityOpenRef`: a React ref tied to a component instance resets on remounts, while direct `store.getState()` access inside an effect is unaffected by remount cycles. The rationale in the PR description is accurate.
- `ResourceWatcher` is only mounted when `isEditorOpen === true`, so `ItemClass.useGet` (and its underlying WebSocket connection) is never active when the pencil button is merely rendered — no unnecessary network requests in Storybook or list-view renders.
- `handleClose` correctly sets `isEditorOpen = false` (unmounting `ResourceWatcher` and tearing down the watch) before calling `Activity.close`, preventing WebSocket leaks when the editor is dismissed.
- Both new commit subjects follow the project convention: `frontend: EditorDialog: …` and `frontend: EditButton: …` — correct area/SubArea/capitalised-verb format, both under 72 characters.
