## Summary

Updated since the last review (2026-05-23): the PR now includes a guard `if keyPart[2] == "" { return }` in `DeleteKeys` that prevents a cluster-scoped cache key from attempting to delete the already-namespace-stripped version of itself (which would be the same key and a no-op at best, incorrect semantics at worst). It also wires `handleKeyGenerationAndDeletion` to call `DeleteKeys` instead of `k8scache.Delete` directly, ensuring informer events clear both the namespace-specific and all-namespace cache entries. Tests are comprehensive and the fix is correct.

### Positive

- `cacheInvalidation_test.go` — `TestDeleteKeys` extended with a "cluster scoped key does not delete namespaced key" case that directly validates the new guard.
