## Verdict: APPROVE

## Summary

Replaces silent Jaeger→OTLP fallback with an explicit rejection error, extracts `validateTracing()` as a helper, adds the `--jaeger-endpoint` flag so the rejection surfaces at startup validation, and updates both config and telemetry tests. The code change is correct and well-tested.

## Findings

### Warnings
- **[MEDIUM]** `backend/pkg/config/config.go` — Commit subject `"backend: Reject unsupported Jaeger tracing endpoint"` is missing the SubArea segment. This change spans both `config` and `telemetry` packages; a good subject would be `"backend: telemetry: Reject unsupported Jaeger endpoint; add --jaeger-endpoint flag"`. Please amend/squash.

### Suggestions
- **[LOW]** `backend/pkg/config/config.go:176` — The retained `"at least one exporter"` check has a pre-existing logic issue: the condition checks `StdoutTraceEnabled != nil && *StdoutTraceEnabled` (stdout IS enabled), which means this block can only fire when stdout is enabled but Jaeger and OTLP endpoints are empty. However, if stdout is enabled you *do* have an exporter, so the error message would be misleading. This bug pre-dates this PR; worth a follow-up issue.

### Positive
- Extracting `validateTracing()` improves readability and is the right refactor before adding Jaeger-specific validation logic.
- New `--jaeger-endpoint` flag with description `"Jaeger collector endpoint (unsupported; use otlp-endpoint)"` surfaces the deprecation clearly to operators at startup — excellent UX.
