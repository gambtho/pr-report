## Verdict: APPROVE

## Summary
Adds an `apiVersion` check alongside the existing `kind` check when resolving owner reference links in `MetadataDisplay.tsx`. Previously, a custom resource sharing the same `kind` as a built-in Kubernetes resource (e.g. a CRD named `Deployment`) would be incorrectly linked to the built-in resource detail page. The fix compares `resourceClass.apiVersion` with `ownerRef.apiVersion`, falling through to plain text when they differ.

## Findings

### Suggestions
- **[MEDIUM]** `frontend/src/components/common/Resource/MetadataDisplay.tsx:69` — The check `resourceClass.apiVersion === ownerRef.apiVersion` uses strict equality. Some resources may represent `apiVersion` in different formats (e.g. `"v1"` vs `"core/v1"`) depending on how the resource class was registered. While in practice this should be consistent, a brief comment noting that the comparison relies on format parity would help future maintainers.

### Positive
- Minimal, well-targeted fix. Refactoring the repeated `ResourceClasses[ownerRef.kind as keyof typeof ResourceClasses]` lookup into a `const resourceClass` variable also reduces redundancy. Plain-text fallback for mismatched `apiVersion` is the correct behavior.
