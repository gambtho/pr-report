## Verdict: NEEDS_DISCUSSION

## Summary
Adds `@typescript/native-preview` (the experimental Go-rewrite TypeScript compiler, `tsgo`) as a dev dependency and replaces the `"tsc": "tsc"` script with `"tsc": "tsgo"`. Also changes `tsconfig.json` `moduleResolution` from `"node"` to `"bundler"` and excludes the package from the plugin dependency sync script. The PR claims a significant type-check speed improvement (18–20 s → 3–4 s).

## Findings

### Warnings
- **[HIGH]** `frontend/package.json` — `@typescript/native-preview` is pinned to a **nightly dev snapshot** (`7.0.0-dev.20260519.1`). Nightly packages can be unpublished, have breaking type-checker differences from stable TypeScript, or silently accept code that `tsc` rejects (or vice versa). Using a nightly as the sole `npm run tsc` / CI type-checker is a reliability risk. Consider keeping the official `typescript` package for CI and using `tsgo` as an opt-in developer alias (e.g. `"tsc:fast": "tsgo"`).
- **[MEDIUM]** `frontend/package.json:123` — Replacing `"tsc": "tsc"` with `"tsc": "tsgo"` means all CI jobs and contributor workflows that call `npm run tsc` now use the experimental compiler. Any `tsgo`-specific quirk (missing flags, different strictness, module resolution differences) becomes a silent breakage in the CI pipeline.
- **[MEDIUM]** `frontend/tsconfig.json` — Changing `moduleResolution` from `"node"` to `"bundler"` affects how imports without extensions are resolved. While Rsbuild uses bundler resolution, VS Code and other IDE TypeScript integrations typically use the locally installed `typescript` package (not `tsgo`), creating a discrepancy between IDE and CLI type-checking. Downstream plugin authors using the same tsconfig might also be affected.

### Suggestions
- **[LOW]** `frontend/package.json` — The semver range `"^7.0.0-dev.20260519.1"` will not auto-upgrade to newer dev snapshots (npm's caret is conservative with pre-release tags). Document this explicitly in a comment or PR description so maintainers know manual updates are required.

### Positive
- The 5× speedup for local development is compelling, and `tsgo` is the official Microsoft-authored successor to `tsc`. The approach of excluding the package from plugin dependency sync is correct.
